# PR #127 Design: Companion Action History Pruning

## Summary
Implements companion action history pruning to cap the `companion_actions` table at a configurable limit (default 50 entries per companion). Also adds `companion_joined`, `companion_departed`, `companion_acted` event types to the world event system.

## Problem
Companion action history was growing indefinitely in the `companion_actions` SQLite table, with no cap or cleanup mechanism.

## Goals
- Cap companion action history at configurable limit (default 50)
- Prune oldest entries when limit is exceeded
- Preserve the most recent entries
- Summarize pruned entries as JSON metadata (not prose)
- Protect `history_summary` audit records from being re-pruned
- Remove `buildDigest` dead code

## Changes
- **`packages/backend/src/story/story_compactor.ts`**: Added `compactCompanionHistory()` function. Removes `buildDigest()` dead code. Protects `history_summary` records from pruning cycles. `generateActionSummary()` returns JSON metadata, not prose.
- **`packages/backend/src/storage/sqlite_repository.ts`**: Integrated `compactCompanionHistory()` into `SQLiteCompanionRepository.pruneCompanionHistory()`. Runs in a transaction: deletes pruned IDs, inserts a `history_summary` record.
- **`packages/backend/src/storage/companion_pruning.test.ts`**: Integration tests for pruning behavior with real SQLite (191 lines added).
- **`packages/backend/src/world/companion_events.test.ts`**: Unit tests for companion event generation (91 lines added).
- **`packages/backend/src/world/events.ts`**: Added `companion_joined`, `companion_departed`, `companion_acted` to `CompanionEventType`. Added entries to `companionActionEventMap`.
- **`packages/backend/src/world/types.ts`**: Extended `WorldEvent.type` union with `companion_joined`, `companion_departed`, `companion_acted`.
- **`.beads/issues.jsonl`**: Updated issue tracking.

## Key Design Decisions
- `generateActionSummary()` returns JSON metadata — no prose narration
- `history_summary` records are excluded from pruning input to prevent re-deletion
- Pruning runs inside a DB transaction for atomicity
- Default limit: 50 actions; override via `COMPANION_ACTION_LIMIT` env var
- New companion event types follow existing `CompanionEventType` union pattern
