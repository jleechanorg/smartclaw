3-Stage OpenClaw Dev Pipeline

Reference Documentation  ·  Last updated: 2026-04-03  ·  Operational

Background

Before this pipeline, changes to ~/.smartclaw/ (the live gateway config) went straight to production with no automated safety gate. A bad openclaw.json edit could crash the gateway, drop Slack messages, or break all agent sessions. Manual review caught some cases, but not consistently.

The 3-stage pipeline adds automated gates between unvalidated changes and production:

  1. Staging branch — changes land on staging first, not main
  2. Canary — automated health checks against a staging gateway before any promotion
  3. CI gate — portable checks run on every PR, blocking merge on failure

Stage 1 — Staging Branch + Worktree

Files: Git infrastructure (no code)

A long-lived staging branch exists in jleechanorg/smartclaw. ~/.smartclaw-staging/ is a git worktree checked out to this branch. staging-promote.sh merges the updated origin/staging into the local staging worktree before promoting to production.

Why a worktree?

A worktree is tied to the git repo — changes in ~/.smartclaw-staging/ can be committed back to the staging branch. A plain directory requires manual sync and loses this traceability.

Creating the staging worktree (done once)

git fetch origin staging
git branch --set-upstream-to=origin/staging staging 2>/dev/null || \
  git branch staging origin/staging 2>/dev/null || true
git worktree add ~/.smartclaw-staging origin/staging

Stage 2 — Auto-Promote

File: scripts/staging-promote.sh

Runs the full 6-point canary against the staging gateway. If all checks pass, merges staging → main and pushes to origin.

Key guards

  • Refuses to promote if staging is not a git worktree
  • Refuses to promote if staging worktree is not on the staging branch
  • Aborts any in-progress merge before starting (idempotent)
  • Stashes uncommitted local changes before merging

Fail-closed

Any canary failure = no promotion. The script exits non-zero and prints the failing check.

Usage

# After merging a PR to the staging branch
bash scripts/staging-promote.sh

# Verify prod health after promotion
bash scripts/staging-canary.sh --port 18789

Stage 3 — CI Gate

File: .github/workflows/staging-canary-gate.yml

Fires on every PR. Detects whether openclaw.json changed in the diff. If so, runs the full canary (which also reads package.json for Check 5 — SDK protocol version). package.json changes alone do not trigger the canary.

Canary checks — CI vs local

CheckWhat it doesRuns in CI?
1. Gateway health /health endpoint responds No — needs local gateway
2. Config schema JSON valid, no crash-keys, critical keys present Yes
3. Native module ABI better-sqlite3 loads with correct Node No — needs local Node + extensions
4. Slack token validity apps.connections.open succeeds No — needs live gateway
5. SDK protocol version @agentclientprotocol/sdk ≤ 0.16 Yes
6. Heartbeat latency /health responds in < 5s No — needs local gateway
Portable CI: GitHub-hosted runners execute checks 2 and 5 only. Checks 1, 3, 4, and 6 need your machine — use Stage 3b (self-hosted) or run staging-canary.sh manually.

Stage 3b — Full canary in CI (self-hosted)

File: .github/workflows/staging-canary-full.yml

When openclaw.json changes in a PR, runs scripts/staging-canary.sh (all 6 checks) on a self-hosted runner (e.g. Mac with ai.smartclaw.staging on port 18810).

Setup: .github/SELFHOSTED_TEST.md — register a runner with labels self-hosted and openclaw-staging. Optional variables: STAGING_CANARY_PORT, OPENCLAW_STAGING_CONFIG. Set ENABLE_STAGING_CANARY_SELFHOSTED=false to skip when the runner is offline.

Not included: monitor-agent.sh remains on LaunchAgent; it is not a PR gate.

The Full Canary (6/6 Checks)

File: scripts/staging-canary.sh

Run against the staging gateway before merging staging → main, or rely on Stage 3b if configured. This is the final gate before production promotion.

# Start staging gateway (one-time or after machine reboot)
bash scripts/staging-gateway.sh start

# Run full canary
bash scripts/staging-canary.sh --port 18790

Gateway management

CommandWhat it does
staging-gateway.sh startStart staging gateway on port 18790
staging-gateway.sh stopStop staging gateway
staging-gateway.sh statusShow PID, port, health status

The staging gateway is a separate process from production (port 18789). It shares the same binary and Node.js but uses ~/.smartclaw/staging/openclaw.json as its config.

Day-to-Day Workflow

Normal PR flow (gateway config changes)

# 1. In feature worktree: edit the file (openclaw.json, agents/, skills/, SOUL.md, etc.)

# 2. Open PR targeting: staging (NOT main)
gh pr create --base staging --head feat/my-change

# 3. CI gate fires automatically
#    If it fails: fix the config issues, push

# 4. Merge PR to staging
#    staging-promote.sh merges origin/staging into the local staging worktree

# 5. Run canary against staging gateway
bash scripts/staging-canary.sh --port 18790
#    If it fails: investigate, fix in a new PR

# 6. Promote staging → main
bash scripts/staging-promote.sh

# 7. Production picks up the change via git pull; if needed: stop && start the gateway

Non-config changes (docs, scripts, Python source)

PR → main directly. CI runs as normal. No staging gate required.

Known Gaps

orch-1ps.4 — Git hooks for auto-restart P1 — Open

What was intended: Post-merge or post-receive hooks in ~/.smartclaw-staging/.git/hooks/ that automatically restart the staging gateway when the staging branch is updated.

Current workaround: Run manually after staging promotion:

bash scripts/staging-gateway.sh stop
bash scripts/staging-gateway.sh start

What building this would involve:

  • A post-merge or post-receive hook in ~/.smartclaw-staging/.git/hooks/
  • The hook detects a git pull or merge in the staging worktree
  • Calls bash scripts/staging-gateway.sh stop && bash scripts/staging-gateway.sh start

Staging Sync Caveat

The staging branch must stay in sync with (or ahead of) main. Never let it fall behind.

A bug in the initial implementation created the staging branch from a stale commit (3 days behind main). Promoting from stale staging would have been a regression — staging would overwrite newer main changes.

Verify staging is current

git rev-list staging..main --count

Non-zero output = staging is behind main. Fast-forward:

git checkout staging
git merge --ff-only origin/main
git push origin staging
